Method of Communication

Our main form of communication software is Discord. We will have different channels for development, casual conversations, etc. by using Facebook Messenger and general texting. Through these channels we will be able to keep everything in one place while making sure our work is organized.

Within 30min - 1hr regardless of the method of communication, unless it has been communicated earlier that someone will not be able to respond within this time frame for a legitimate reason.

Coding Tools and Software

We will be using Trello for consolidating our tasks and keeping track of our current progress. For version control, we will be using GitHub since it is widely used by many companies and developers. As for creating UML diagrams, we will be using the Eclipse UML generator.

Code Standard and Version Control

Only code that aligns with our DOD (see below) should be committed to our GitHub repository.

Deliverable Proofreading

Before submitting the final version of the deliverable, all members will look over and proofread all relevant data to ensure that there are no grammar/punctuation mistakes, meet the assignment requirements, and have satisfactory writing. This must be done atleast one day prior to deliverable deadline so the necessary alterations can be made.

Dividing Up Work

During the initial deliverable introduction, we will sit down and get to know everyone’s strengths and weaknesses and then appropriately divide up the tasks amongst ourselves according to the requirements of the deliverable. Everyone must commit at least 5 hours of their time every week working on this project. This is justified due to everyone in the group being a full- time student, with some having part-time jobs, and to have enough time for work for other courses.

Resolution For Conflicts

If a member is behaving inappropriately, missing madatory meetings too much, is not being responsive, does not contribute, or is agreed by other members that they are slacking with their contribution, they will be contacted to see if they are doing ok and if they need any assistance. If they do not respond then a warning will be issued. The warning is that if they do not step up and do the work to the best of their ability, they will be reported to the TA/Professor about their behaviour/contribution and, if justifiable, will be kicked from the team. However, we hope this will never happen during the duration of this course.

Definition of Done

For A Feature

The relevant code functions as intended, does not cause bugs elsewhere, is commented, and documented. This is as a result of the following:

  • All members briefly tested the feature, reviewed the style of coding and documentation to be as consistent as possible with the rest of Matplotlib, and gave their consent for submission.
  • Designated member(s) tested Matplotlib with the new feature rigorously via pytest and recorded testing results.
  • If a test case failed, code was debugged, re-written, and re-tested.
  • If the changes as a result of debugging were extensive, code was reviewed by the group again.
  • This process was repeated until the group gave their consent for submission.

For A Deliverable

The deliverable meets the assignment requirements. This is as a result of the following:

  • All members proofread the deliverable for meeting the assignment requirements and for satisfactory writing.
  • Proposed changes were discussed on Discord and implemented.
  • If the changes as a result of proofreading were extensive, the deliverable was proofread again.
  • This process was repeated until the group gave their consent for submission.
